# Matryca Plumber v1.8 — Edge Computing & Performance Update

**Status:** Shipped in **1.8.0** (see [`CHANGELOG.md`](../CHANGELOG.md) `[1.8.0]`).  
**Strategic constraint:** v1.8 adds **zero** new semantic or graph-manipulation features — only performance, memory governance, and host-survival mechanics.

**USP:** Flawless, silent, indefinite operation on a **CPU-only 16 GB laptop**, indexing **≤10,000** Markdown pages without freezing the host OS.

**Engineering spec:** [`openspec/llm-performance.md`](openspec/llm-performance.md)  
**Architecture context:** [`ARCHITECTURE.md`](ARCHITECTURE.md#edge-computing--performance-v18)  
**Maintainer narrative:** [`PROJECT_DIARY.md`](PROJECT_DIARY.md) (2026-05-27 entry)

---

## Why v1.8 exists

Phase 14d shipped the **Context Acceleration Shield** (summaries + content-before-task ordering). Production graphs at thousands of pages exposed three remaining bottlenecks:

1. **KV-cache misses** — Bootstrap harvest still sent `Task` before `Content`; alias maps lived in the **system** prompt; cognitive modules rebuilt unrelated prefixes per page.
2. **RAM growth** — Full BM25 token lists, unbounded semantic cache RAM, and full-graph backlink rescans on every bootstrap.
3. **Host stalls** — Tight synchronous loops over 10k files with no cooperative yield and aggressive daemon checkpoint writes.

v1.8 closes that gap without changing Logseq semantics or adding new lint modules.

## Environment variables

Full list and defaults: [`.env.example`](../.env.example) section **v1.8 Edge computing & performance**.

| Variable | Default | Purpose |
|----------|---------|---------|
| `MATRYCA_RAM_BUDGET_MB` | `4096` | Soft RSS budget; warnings in logs when exceeded |
| `MATRYCA_ALIAS_PROMPT_MAX_CHARS` | `2048` | Cap AliasIndex footer in stable user block |
| `MATRYCA_SEMANTIC_CACHE_MEMORY_ENTRIES` | `512` | In-process semantic inference LRU |
| `MATRYCA_BM25_MODE` | `resident` | `resident` or `ondemand` (rebuild per query) |
| `MATRYCA_BOOTSTRAP_YIELD_EVERY` | `25` | `yield_host()` interval during harvest |
| `MATRYCA_BOOTSTRAP_CHECKPOINT_EVERY` | `50` | Daemon state persist interval during Phase 1 |
| `MATRYCA_BOOTSTRAP_IO_BATCH_PAUSE_MS` | `2` | Micro-pause after non-LLM harvest steps (not thermal — see below) |
| `MATRYCA_YIELD_SLEEP_MS` | `0` | Extra yield sleep on `yield_host()` ticks (use 1–5 on HDD) |
| `MATRYCA_LLM_CLUSTER_HISTORY` | `false` | Full vs minimal cluster ack in Ermes history |
| `MATRYCA_LLM_PROMPT_CACHE_MODE` | `off` | Reserved: future server slot integration |
| `MATRYCA_CPU_SANDBOX` | `true` | CPU affinity + idle I/O class when `psutil` is installed (`[edge]` extra) |
| `MATRYCA_PLUMBER_CPU_AFFINITY` | *(unset)* | Optional comma-separated core list; overrides auto sandbox mask |
| `MATRYCA_GRAPH_READ_MMAP` | `true` | Phase 1 regex catalog path uses `mmap_graph_page()` |
| `MATRYCA_MMAP_MIN_BYTES` | `4096` | Skip mmap for tiny files (read into heap instead) |

Existing thermal and low-priority knobs remain: `MATRYCA_THERMAL_DELAY_*`, `MATRYCA_PLUMBER_LOW_PRIORITY_MODE`.

Install CPU sandbox support: `uv sync --extra edge` or `pip install matryca-plumber[edge]` (`psutil`).

### Thermal pauses vs cooperative micro-yield

| Sleep source | Typical duration | When |
|--------------|------------------|------|
| **Thermal** (`MATRYCA_THERMAL_DELAY_BOOTSTRAP` / `_COGNITIVE`) | ≥ 1 s (operator-tuned) | After each **LLM** turn in bootstrap or cognitive pipeline |
| **Micro-yield** (`MATRYCA_BOOTSTRAP_IO_BATCH_PAUSE_MS`, default 2 ms) | &lt; 1 s | Between non-LLM harvest steps so the host OS stays responsive |
| **`yield_host()`** | `MATRYCA_YIELD_SLEEP_MS` (often 0) | Every `MATRYCA_BOOTSTRAP_YIELD_EVERY` files during harvest |

Tests that assert thermal behavior must filter `time.sleep` calls with `s >= 1.0` so micro-yields are not counted as thermal delays (see `test_bootstrap_harvest_thermal_delay_only_after_llm` in `tests/test_master_catalog.py`).

Software-edge directives (frozen KV prefix, adaptive LLM, mmap): [`v1.8-SOFTWARE-EDGE-PLAN.md`](v1.8-SOFTWARE-EDGE-PLAN.md).

## Operator workflows

### Large vault bootstrap

1. Point `LOGSEQ_GRAPH_PATH` at a **clone** of your graph (see README).
2. Enable v1.8 env block from `.env.example`.
3. Start daemon: `matryca plumber start --foreground` — Phase 1 harvest yields periodically; watch Sovereign UI progress.
4. After Phase 1: memory teardown runs automatically; Phase 2 begins with cluster map warm.

### Synthetic load testing

```bash
uv run python scripts/gen_synthetic_graph.py /path/to/test-graph --count 1000
uv run matryca plumber start --foreground
make perf   # slow memory/soak tests (--no-cov; not subject to the 70% suite gate)
```

## Explicit non-goals (v1.8)

- New cognitive modules or graph algorithms.
- Louvain / clustering quality changes.
- Cloud LLM orchestration or multi-GPU scheduling.
- Embedding-based search replacing BM25 (unless internal-only with identical query API).
